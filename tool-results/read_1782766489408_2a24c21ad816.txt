     1→# Executive Summary
     2→
     3→**Российский футбольный драфт-симулятор** – это проект веб-игры по мотивам [38‑0](https://38-0.app) для Telegram Mini Apps. Игрок строит **идеальную сборную всех времён** из игроков российской Высшей лиги (РФПЛ/РПЛ) и симулирует полный сезон. Аналогично оригиналу 38‑0, каждая «попытка» состоит из последовательности «бросков колеса» (Крутить), выбора клубов и игроков, наполнения состава и мгновенного проигрыша сезона. Цель – добиться максимального результата (теоретически «38‑0» – 38 побед в 38 матчах, 114 очков).
     4→
     5→Проект ориентирован на российских болельщиков (интерфейс и тексты – полностью на русском языке). В качестве **источника данных** предполагается история советского и российского футбола (сезоны высших лиг с 1990 по 2026 год). Будет использоваться **Telegram Mini App** для запуска в мессенджере (через WebApp SDK) с учётом официальных рекомендаций Telegram (авторизация, безопасность, кнопки, темы и т.д.). Архитектура проекта — **full‑stack JavaScript/TypeScript** (например, React/Next.js на фронтенде и Node.js/NestJS на бэкенде) с базой данных PostgreSQL (или Supabase) и Redis. Разработку планируется вести через генерацию кода (Mimo Code, GitHub Copilot и др.) по заранее составленному подробному техническому документу (PRD/GDD/TDD).
     6→
     7→В результате получим технический документ в формате Markdown, включающий **Executive Summary, PRD, GDD, TDD**, планы поэтапной разработки, бэклог, чек-листы, ER-диаграмму, спецификацию API (OpenAPI) и roadmap. Документ будет достаточен для начала кодирования проекта практически без дополнительных уточнений.

     9→# Product Requirements Document (PRD)
    10→
    11→## Цели и задачи
    12→
    13→- Создать мобильную и desktop‑версии игры-драфта **“30-0 RPL”** по образцу 38‑0, но с **российским футбольным контентом**.  
    14→- Поддержка исторических данных: включить все сезоны высшего дивизиона (СССР/Россия) с **1990 по 2026 год**.  
    15→- Использовать формат **Telegram Mini App**: игра запускается внутри Telegram (с ботом), авторизация через Telegram Login, хранение данных (сессии) – по стандартам WebApp.  
    16→- Интерфейс: **минимализм, тёмная тема, русскоязычный текст, Apple-стиль, Telegram-стиль**, использование эмодзи Apple.  
    17→- Игровая механика: драфт по схеме, как в 38‑0: рандом-клуб+сезон на каждом шаге, выбор игрока в нужную позицию, заполнение состава из 11 человек, симуляция сезона.  
    18→- Режимы игры:
    19→  - **Классический**: рейтинг игроков виден, обычная сложность.  
    20→  - **Тяжелый уровень**: рейтинги игроков скрыты (игрок видит только характеристики/позицию, без очков рейтинга).  
    21→  - **Один клуб XI**: выбрать один клуб и собрать его лучших игроков из разных лет.  
    22→  - **Челлендж**: одинаковые начальные условия для всех игроков (одна схема, выпадение одного и того же начального клуба/сезона).  
    23→  - **PvP/Лиги**: соревнования с друзьями или глобальные турниры по тем же правилам (см. «Leagues» на сайте 38‑0).  
    24→
    25→- Ограничения: каждая «попытка» использует не более 10 *reroll* в день (бесплатные новые выборы вместо текущего) – по аналогии с оригиналом.  
    26→- Социальные функции: **шаринг** результата и состава в Telegram (поделиться ссылкой или картинкой), **лидерборд** (глобальный и друзей) на основе набранных очков (подчётов матчей), **достижения** (например, «38‑0-0», «Чемпион с разницей +50», «Набрать 100+ очков»), Пригласить друга и заработать дополнителные 10 рероллов.
    27→- **Монетизация**: без Pay-To-Win. Главное – **не давать преимуществ** в силе команды за деньги.

    29→## Пользовательские сценарии
    30→
    31→1. **Регистрация/Авторизация:** пользователь заходит в Telegram, запускает бот/мини‑приложение через WebApp. При первом запуске предъявляет разрешение боту (InitData), данные проходят проверку подписи. На входе игра предлагает ввести имя (или брать Telegram ник) для лидеров и статистики.
    32→2. **Главное меню:** выбор режима (классика, Тяжелый уровень, Один клуб, Челлендж, PvP/League). Отображение сохранённых результатов, статистики, достижений. Кнопки: начать новую игру, история игр, настройки.
    33→3. **Выбор схемы:** перед драфтом выбор расстановки (4-3-3, 4-4-2, 3-5-2, 3-4-3, 4-2-3-1 и т.д.). После выбора кнопка «Старт».
    34→4. **Процесс драфта:** 
    35→   - Нажатие **«Крутить»**: «колесо» выбирает рандомно клуб и сезон (из доступного периода). На экране показывается: сезон (год), прогресс заполнения команды.  
    36→   - Появляется **список игроков** клуба/сезона (поля: имя, позиции, рейтинг). Возможно фильтрация/сортировка (по позиции или рейтингу). Режим Тяжелый может не показывать рейтинг (заменить звездочками или скрыть).  
    37→   - Игрок выбирает одного футболиста. Этот игрок фиксируется в составе на ближайшую незаполненную позицию этого типа (например, левый защитник, центральный нападающий и т.д.).  
    38→   - Повторять «Крутить + выбор игрока» до заполнения всех 11 позиций. Допустимо менять ранее выбранных игроков местами (Drag & Drop по интерфейсу) до окончания драфта. Штраф за размещение игрока не на основной позиции: если, например, в защите случайно оказался форвард, это уменьшает силу этого сегмента (описано в модели симуляции ниже).
    39→   - **Реролл:** в любой момент, пока драфт не завершён, игрок может нажать «Reroll» (до 10 раз в сутки бесплатно), чтобы пропустить текущий клуб/сезон и получить новый (колесо крутится повторно без выбора игрока).
    40→5. **Симуляция сезона:** когда XI заполнен, нажать «Симуляция». Движок симуляции мгновенно проигрывает все 30–34 игры (для РПЛ/Высшей лиги) с учётом силы состава. Возвращается итог: таблица с очками (3 очка за победу, 1 за ничью), позиция команды, количество побед/ничьих/поражений, голов забито/пропущено, MVP (лучший игрок по системе) и количество очков (макс 114). Показываются достижения (например, «Непобедимый сезон»). Система позволяет повторно проиграть этот же XI (с новой серией случайностей).
    41→6. **Результат и соцфункции:** после симуляции показывается экран результата, с возможностью поделиться в Telegram (в чат или канал) карточкой результата (изображение с эмблемами, бомбардиром, очками). Также результат сохраняется в профиле пользователя и влияет на глобальный/дружеский лидерборд.
    42→7. **Профиль:** в боте/приложении есть раздел «Профиль», где видны история игр пользователя, лучшая сборная, статистика (суммарные очки, выигрышные серии, избранные клубы/игроки), достижения, топ-результаты друзей.
    43→8. **Админ-панель:** закрытая зона (веб-интерфейс для админа) – редактирование и обновление базы данных игроков/рейтингов, настройка ежедневного челленджа, управление контентом (главная страница, инфо).

    45→# Game Design Document (GDD)
    46→
    47→## Механика драфта
    48→
    49→- **Формирование команды:** игрок выбирает формуцию из списка (4-4-2, 4-3-3, 3-5-2, 5-3-2, 4-2-3-1, 3-4-3 и др.). Система задаёт слоты (GK, CB×2, LB, RB, CM×х, LM/RM, etc.) в этой схеме.
    50→- **Спин-колесо:** при каждом раунде генерируется случайный клуб и сезон. Реализуем **исключение «мертвого» спина** – система гарантирует, что выбранный клуб/сезон имеет хотя бы один незаполненный позиционный слот, чтобы игрок мог выбрать игрока. Формально: колёсный рандом выбирает из множества подходящих (которые соответствуют хотя бы одному пустому слоту).
    51→- **Драфт игрока:** игрок видит список всех футболистов выбранного клуба в том сезоне (основная позицая и дополнительные, рейтинг). Игрок кликом добавляет футболиста в команду (в наименьший подходящий слот). Одна и та же персона не может быть выбрана дважды (даже из разных сезонов, если это тот же игрок – надо в базе хранить ID футболиста).
    52→- **Позиционная логика:** каждый футболист имеет «основную» позицию и ряд дополнительных. Если его сажают не по основной, он считается играющим в «не своей зоне». В симуляции это приведёт к штрафу (снижение эффективности, как будто «игра не на месте»).
    53→- **Заполнение состава:** повторять вращение колеса и выбор игрока, пока нет свободных слотов. Допускается менять позиции/игроков между слотами (Drag&Drop). Если игрок передумал, может убрать футболиста из слота (иконкой крестика) и выбрать другого.
    54→- **Реролл:** до 10 раз в сутки – замена текущего раунда (колесо крутится заново с тем же прогрессом).
    55→- **Отображение рейтингов:** игрок видит рейтинг (например, число до 100) каждого футболиста. В режиме Hard эти числа скрыты (только звёздочки или полоска прогресса).
    56→- **Ограничения:** по времени нет таймера, игра не мультиплеерная в реальном времени, но сохранение прогресса.

    58→## Режимы игры
    59→
    60→- **Классика (Normal):** видны все рейтинги, обычное везение.  
    61→- **Без рейтингов (Hidden):** как в Hard – рейтинг скрыт на этапе выбора (пользователь оценивает игрока фамилии и позиции, не зная цифр).  
    62→- **Тяжелый уровень:** аналог «Без рейтингов», плюс усложнённые условия (например, меньше «удачи» при симуляции, более строгие штрафы или повышенная требовательность к балансу команды).  
    63→- **Один клуб XI:** игрок выбирает один клуб (например, «ЦСКА»), и драфт ограничен только футболистами, выступавшими за этот клуб на протяжении истории. Меняется интерфейс: вместо колеса на каждом шаге – только сезон (колесо не меняет клуб). Это позволяет собрать «лучших игроков одного клуба», но с учётом позиций и баланса.  
    64→- **Челлендж:** один режим на день, фиксированные условия (например, одна случайная стартовая схема + первые три драфта заранее выбираются и публикуются). Все игроки соревнуются в одинаковом испытании.  
    65→- **Лиги / PvP:** система групповых игр, где друзья или члены группы играют «той же командой» (похожими условиями), и сражаются за очки лидерборда.

    67→## Интерфейс (UX/UI)
    68→
    69→- **Главный экран:** большие карточки/кнопки режима (Классика, Тяжелый, и тд.), показ свежего результата/челленджа, кнопка профиля, шестерёнка (настройки).  
    70→- **Экран драфта:** разделённый вид – слева «колесо» (показывает клуб и сезон), справа текущий состав (сеткой или списком слотов GK, DEF, MID, FWD). Ниже – список доступных игроков (карточки). Кнопки: «Крутить», «Реролл (хN)». Цветовая палитра тёмная (фон #1E1E1E, яркие акценты для кнопок). Apple Emoji используются в тексте (⭐️, ✅, 🎯).  
    71→- **Компоненты:** 
    72→  - Кнопки (основные круглые или с плавными углами), переключатели схем. 
    73→  - Skeleton Loading (серые полосы) при загрузке данных (списка игроков или таблицы). 
    74→  - Empty State (например, «нет сохранённых игр»). 
    75→  - Error State (если не удалось получить данные, подсказка «проверьте интернет»). 
    76→  - Success State (краткое всплывающее «Успех!» после совершения действия, например, успешный reroll).
    77→- **Адаптивность:** в мини-аппе Telegram доступна мобильная версия с учётом SafeArea (вырезы экрана), главное меню и игра должны хорошо выглядеть в портрете и ландшафте. Интерфейс «размазывается» по экрану 100% ширины. В десктопе (Telegram Desktop) окно шире, можно показывать боковую панель с лидерами или чат.
    78→- **Стиль:** вдохновение – [38-0](https://38-0.app) (минималистичный, монохромные фоны, акцент на эмодзи и крупном контенте), Apple Human Interface (чистые шрифты, отступы). Цвета: основной – тёмно-серый/чёрный, акценты – красный (для выделения, например, забитых голов), зелёный (для побед), серебряный/золотой (для достижений). Типография – шрифт без засечек (например, SF Pro или аналог), размеры регулируемые под мобильный.

    96→## Сущности и данные
    97→
    98→- **Игрок (Player):** уникальный футболист (имя, фамилия, ID). Хранить базовую информацию (Гражданство, текущий клуб в регионе). 
    99→- **Клуб (Club):** футбольный клуб. Атрибуты: имя, страна.  
   100→- **Сезон (Season):** календарный сезон (например, «2001/02»). Для каждого сезона и клуба – отдельная запись (ClubSeason) с итоговым местом, статистиками клуба.  
   101→- **PlayerSeason (профиль игрока в сезоне):** указывает, за какой клуб играл игрок в данном сезоне (FK), его возраст. Ссылка на таблицу PLAYER. Один игрок может иметь множество сезонов (один за другим).  
   102→- **Рейтинг (Rating):** оценка силы игрока в конкретном сезоне. Можно хранить в PlayerSeason (например, число 0-100). Источник: агрегированные данные (см. ниже).  
   103→- **Позиция (Position):** основная и дополнительные позиции игрока (например, «ЦЗ», «ЛП» и т.д.).  
   104→- **Пользователь (User):** зарегистрированный игрок мини-аппа (Telegram User ID, имя, статистика).  
   105→- **Запуск игры (Run):** одна «попытка» пользователя; содержит выбранную схему, дату, список из 11 PlayerSeason (состав), режим игры, список применённых reroll, итоговые данные (очки, победы и т.п.).  
   106→- **Результаты матчей (MatchResult):** при симуляции генерируются результаты (каждая игра: домашняя/гостевая команда, счёт). Могут храниться при необходимости, но для MVP/статистики можно считать на лету.  
   107→- **Достижение (Achievement):** название, описание, условие (например, «30-0» – 114 очков, «Победитель чемпионата» – первое место и титул, «Без поражений» – 0 поражений). Связь с User (получил или нет).  
   108→- **Лидерборд:** хранится кэш (или вычисляется on-the-fly) топовые RUNs по очкам (глобальный и среди друзей).  
   109→- **Данные исходов:** необязательные - можно генерировать динамически.

   111→## Источники данных
   112→
   113→Для наполнения базы потребуется собрать:
   114→- **Списки игроков и составы клубов:** для каждого сезона и каждого клуба – полный список футболистов.  
   115→- **Рейтинги:** числовая оценка силы игрока в сезоне (от 0 до 100, где 90+ – топ-игроки, см. [44†L130-L134]).  
   116→- **Позиции:** основная позиция, несколько дополнительных (сокращения).  
   117→- **Клубы:** список всех клубов, участвовавших в высшем дивизионе (в СССР-Вышке/Высшей лиге/РПЛ)
   118→
   119→**Источники:**  
   120→- **Transfermarkt** (топ): охватывает всё (94%). Имеет данные по игрокам, переходам и, главное, исторические составы и рейтинги (рыночная стоимость, игровые данные). Минусы: нет официального открытого API (нужен парсинг страницы или внешние сервисы). Доступны пользовательские лиги (часть контента). Лицензия: данные подлежат авторским правам, использование требует осторожности (можно парсить, но надо уважать robots.txt и не превышать лимиты).  
   121→- **WorldFootball.net**: качественный открытый ресурс с полными составами и статистикой по сезонам, матчам, топ-голам. Широко известен, охват большинства стран. Минусы: нет API, есть веб-версия, можно парсить (строгая лицензия нет, контент free).  
   122→- **Wikipedia (ru, en)**: есть общие статьи по сезонам, топ-списки (бомбардиры, чемпионы). Плюс: API, CC BY-SA лицензия (нужна проверка атрибуции). Минус: неполнота, ошибки, разные форматы, надо чистить.  
   123→- **Wikipedia ФК (ru)**: для составов команд, можно парсить списки игроков по клубам и сезонам.  
   124→- **WhoScored/Opta/FBRef/StatsBomb**: снабжают расширенной аналитикой (xG, точные позиции, потенциал и т.д.) для некоторых сезонов, но скорее специализируются на актуальном. Для исторического (1990–2010) данных почти нет. WhoScored (iFrames на сайте, сложен парсинг), FBRef (есть разделы, но частично). Хотя [15] указывает, что Transfermarkt ставит rating как «based on public data» без указания прямых источников.  
   125→- **Каггл и датасеты:** есть данные по некоторым турнирам, но в основном EPL/NBA. Русских данных мало.  
   126→- **Футбольная база (football-database.com)**: платный API, не уверен в охвате.  
   127→- **Официальные сайты РПЛ и архивы:** в основном последние сезоны, разные форматы, не охватывают весь период.  
   128→- **Фанатские архивы и форумы:** могут дать составы (например oldfootball.ru, wildstat.ru), но источники разрознены.

   143→### Сбор и нормализация данных
   144→
   145→План **ETL-пайплайна**:
   146→1. **Сбор (Extract):**  
   147→   - Использовать API/скрапинг Transfermarkt (неофициальные) или [42] доработку, чтобы выкачать данные по сезонам (игроки, статистика). Альтернатива: парсить WorldFootball.  
   148→   - Парсинг Wikipedia (через MediaWiki API) для получения данных (чемпион, финальная таблица, лучшие бомбардиры по сезону).  
   149→   - Если есть данные из футбольных симуляторов (например, рейтинги из Football Manager 2026, которые доступны через сообщество), использовать как референс.  
   150→   - Собрать изображения: эмблемы можно найти на **Wikipedia** (медиафайлы свободны) или через **Wikimedia Commons** (имеют лицензии); фото игроков – не используются (в тексте можно не хранить, для UI достаточно имя + emoji).
   151→2. **Очистка (Transform):**  
   152→   - Убрать дубликаты (разные написания имен, одинаковые игроки). Использовать ID (например, Transfermarkt ID или Wikidata).  
   153→   - Нормализовать названия клубов (в разное время один клуб мог меняться: «Спартак-ЗИЛ» -> «Спартак М» и т.д.) – создать таблицу соответствий.  
   154→   - Нормализация рейтингов: если данные берутся из разных источников (например, Transfermarkt market value, FM overall, условные баллы), пересчитать к единой шкале (можно z‑нормализация внутри сезона). Например, взять средний и стандартное отклонение рейтингов в каждом сезоне, привести к общему 0-100.  
   155→   - Для каждой позиции определить коэффициент «игры не на месте»: например, 0.8–0.9 в учёт силы (т.е. если форварда поставить защитником, его вклад снижается на 10-20%).  
   156→3. **Загрузка (Load):**  
   157→   - Заполнение БД PostgreSQL: таблицы PLAYERS, CLUBS, SEASONS, CLUB_SEASONS, PLAYER_SEASON_STATS (игрок-сезон со всеми полями), USER, RUNS, ACHIEVEMENTS и др.  
   158→   - Раз в сезон (или чаще) обновлять данные (например, подтянуть новые рейтинги текущего года).  
   159→   - Версионирование данных: на уровне БД (целые таблицы могут иметь дату обновления), или хранить лог изменений (для возможности отката).

   161→### Алгоритм симуляции (Season Simulator)
   162→
   163→Моделирование матчей должно учитывать **силу состава и случайность**:
   164→
   165→- **Состав:** подсчитываем для каждой команды (игрока) суммарный рейтинг (или несколько рейтингов для защиты, полузащиты, атаки). Например, в команде среднего уровня 40–50 соотносимые силы; сверхкоманда – около 100-110 (для каждого сегмента).
   166→- **Баланс:** важен баланс между линиями: сильные нападающие + слабая защита ведут к пропущенным очкам. Суммарная сила может быть по каждому сегменту (DEF, MID, ATT) – от 0 до, скажем, 120.  
   167→- **Домашнее поле:** хозяева получают небольшой бонус (например, +2 очка за матч по сравнению с выездом).  
   168→- **Сыгранность и химия:** если игроков нет в реальном команде вместе, немного уменьшать эффективность (чтобы послабить кросс-эпохальные сборные).  
   169→- **Случайность:** в каждом матче добавлять рандом-шума (квадратный корень из разницы, или распределение типа Poisson). Например, мощная команда может неожиданно пропустить гол (3% шанс), а слабая выиграть «подвигом» (1%).  
   170→- **Вероятности:** строим вероятностную модель исхода (например, логистическая функция на разницу сил). Например, P(home win) = sigmoid((S_home - S_away + 5 + random)/20). Вычисляем ожидаемые очки: 3*P(win)+1*P(draw).  
   171→- **Голы и MVP:** сколько голов забьёт команда – пропорционально атакующей силе и случайным факторам. MVP – игрок с наибольшим вкладом (можно определить по очкам рейтинга + случайный бонус за гол).  
   172→- **Штрафы:** если игрок выставлен «не на позиции», его эффективность умножается (например) на 0.8. Это закладывает простой штраф механики.
   173→- **Математика:** можно вдохновляться моделями типа Elo или FIFA ratings. Но упрощённо: сравниваем общее rating, меняем на expected goals. Например, ожидаемые голы (xG) = base*10^( (S_home-S_away)/(maxRatingDiff) ). Затем рандомим результат из Poisson.  
   174→- **Результат:** по итогам всех туров строится турнирная таблица, считаются очки. Итог — количество очков (макс 114 в РПЛ, если 38 игр по 3 очка).
   175→
   176→> **Подтверждение:** в 38‑0 используется подобная модель: «движок взвешивает силу и баланс состава за весь календарь». Наш проект перенимает логику, адаптируя к российской системе (30–34 игр, разные коэффициенты).

   180→## Архитектура и стек
   181→
   182→Предлагаемый стек (фокус на бесплатные/low-cost сервисы):
   183→
   184→- **Frontend (Telegram Mini App):** React/Next.js или Vue 3 + Typescript. Использовать [Telegram Web App JS SDK](https://core.telegram.org/bots/webapps) для интеграции (авторизация, кнопки, безопасный доступ к данным). Next.js удобно для SSR/SSG (ускоряет загрузку). Tailwind CSS для быстрой стилизации (тёмная тема).  
   185→- **Backend:** Node.js (например, NestJS) с Typescript. Node имеет множество библиотек для Telegram Bot API, JWT, базы. Серверная часть будет обрабатывать WebApp InitData, верификацию подписи, выполнять API-запросы. Можно развернуть на **Vercel** (Serverless Functions) или **Netlify**, или **Cloudflare Workers** (разница в поддержке WebSocket если нужен, но здесь not needed).  
   186→- **База данных:** PostgreSQL (например, через [Supabase](https://supabase.com) – бесплатный план до 500MB) или DigitalOcean PostgreSQL. Структуру спроектируем ниже.  
   187→- **Кэш:** Redis (Heroku Redis free tier или Redis на Upstash/Heroku) для быстрого leaderboard и сессий.  
   188→- **Хранилище файлов:** Cloudflare R2 (бесплатные 10GB) или AWS S3 (есть бесплатный уровень) для изображений (эмблемы, графика).  
   189→- **CI/CD:** GitHub Actions: автоматическая сборка фронта (деплой на Vercel/Netlify), тесты и проверка кода.  
   190→- **Secrets/Config:** хранить API ключи и токены в GitHub Secrets.  
   191→- **Mimo Code prompts:** можно разбить генерацию кода на этапы: отдельно сгенерировать модели БД, отдельные prompts для API-эндпоинтов, отдельные для UI-компонентов. Использовать уже написанный документ как вводный контекст.  
   192→
   193→Общий flow: пользователь через Telegram запускает MiniApp → Frontend (React) открывается, получает `initData` (содержит user ID, chat ID) → отправляет на Backend для проверки подписи (HMAC по токену) → Backend возвращает JWT с правами пользователя → Frontend сохраняет токен (LocalStorage Secure) и начинает работу. 
   194→
   195→## Telegram Mini App специфика
   196→
   197→- **Авторизация:** используем **WebApp Login** (InitData). После получения initData на фронте, передаём его на сервер для верификации с помощью HMAC-SHA256 (Telegram рекомендует проверять подпись).  
   198→- **Telegram.WebApp SDK:** используем для доступа к кнопкам, безопасному хранилищу (SecureStorage, HapticFeedback), тегам (SafeArea). Например, кнопка MainButton может быть использована для «следующий шаг» при драфте.  
   199→- **MainButton / BottomButton:** можем настроить основную кнопку Telegram (снизу) для действия «Spin» или «Simulate». SecondaryButton (бот API 7.10+) – для сброса или закрытия.  
   200→- **BackButton:** поддержка WebApp.onEvent('backButtonClicked') для навигации по экранам (в историях/настройках).  
   201→- **Темы:** поддержка светлой/тёмной темы, используя `Telegram.WebApp.colorScheme`. Начальная тема – ночная.  
   202→- **Хранилище:** Telegram.WebApp.get/setSettings для состояние (кэш схемы, история) либо использовать `Telegram.WebApp.secureStorage` для сохранения прогресса между запусками.  
   203→- **Безопасность:** обязательно проверять `initData` подпись на сервере (см. [24] L1601-1609). Использовать HTTPS. Защита от **Replay Attack** – проверка поля `auth_date` (не старше нескольких часов). JWT токен сервера – для запросов API (refresh token хранить в secureStorage, минимальный срок действия).
   204→- **Rate Limits:** Telegram Bot API по WebApp – примерно 1000 запросов/сек (не проблема для маленькой игры). Наши бекенды могут попадать под бесплатные сервис-лимиты (например, Supabase – 500 запросов в секунду, Redis 25 ops/s).
   205→- **Deep Links:** поддержать `tg://resolve?domain=YourBot&start=payload` чтобы открывать мини-апп сразу по ссылке с контекстом (например, приглашение в игру).
   206→- **Push Notifications:** Telegram Mini App сам по себе не шлёт пуши; можно интегрировать бота, который через Telegram Bot API шлёт сообщения (но это затрагивает отдельный workflow).

   210→БД на PostgreSQL. Основные таблицы (ER-диаграмма см. выше):
   211→
   212→- **users**: `user_id (PK, Telegram ID)`, `username`, `telegram_name`, `created_at`, `settings JSONB (настройки)`.

   242→## REST API
   243→
   244→Основные эндпоинты (JSON через HTTPS):
   245→
   246→- `POST /auth/telegram` – принять `initData`, проверить подпись, создать/обновить пользователя, вернуть JWT.  
   247→- `GET /clubs` – получить список клубов.  
   248→- `GET /seasons` – список сезонов (1990–2026).  
   249→- `GET /clubs/:clubId/seasons/:seasonId/players` – получить список игроков клуба в сезоне (id, имя, позы, рейтинг).  
   250→- `GET /formations` – получить доступные схемы.  
   251→- `POST /runs` – создать новый **run** (запустить драфт). Тело: user_id (из JWT), formation_id, mode. Возвращает run_id и первый клуб/сезон.  
   252→- `POST /runs/:runId/draft` – выбрать игрока. Тело: player_season_id. Возвращает следующий клуб/сезон.  
   253→- `POST /runs/:runId/reroll` – сброс текущего раунда (если есть доступные reroll). Возвращает новый клуб/сезон.  
   254→- `POST /runs/:runId/simulate` – завершить драфт (если 11 игроков выбрано), прогнать симуляцию, вернуть результат (очки, позиции, goals, mvps).  
   255→- `GET /runs/:runId` – получить детали прошлого Run (состав, статистика).  
   256→- `GET /users/:userId/runs` – список запусков пользователя.  
   257→- `GET /leaderboard` – общая таблица топ-результатов (с параметром ?mode=classic/hidden, ?friends=1 для друзей, ?limit=10).  
   258→- `GET /achievements` – список достижений, `POST /users/:userId/achievements/:achvId` – отметить, если выполнено.  
   259→- **Ошибки:** стандартизовать JSON (code, message). Валидировать все входные данные. Вернуть 401 при ошибках авторизации, 400 при логических ошибках (например, симуляция без полного состава).

   264→paths:

   288→## Безопасность
   289→
   290→- **Подпись InitData:** обязательно проверить HMAC-SHA256 с использованием токена бота.  
   291→- **JWT & Refresh:** использовать JWT для API, короткий срок жизни (1 час) + refresh tokens (через безопасное хранилище).  
   292→- **OWASP:** защита от SQL-инъекций (использовать ORM/парам. запросы), XSS (валидация данных, Content Security Policy для фронта), CSRF не критично (одностраничка, но можно токен).  
   293→- **Rate Limit:** ограничить API (например, 100 req/min или по IP).  
   294→- **Replay Attack:** проверять `auth_date` (в initData) – не допускать, если старая подпись.  
   295→- **Хранилище:** если используются Cookies, пометить Secure, HttpOnly.  
   296→- **Протоколы:** HTTPS везде.  
   297→- **Прочее:** регулярное обновление зависимостей, использование GitHub Dependabot.
